# AIMXS Customer-Hosted Air-Gapped Installation and Update

Last updated: 2026-03-02

## Purpose

Define a repeatable install/update path for customer-hosted AIMXS deployments where policy decisions stay local and no external data egress is required.

## Scope

- Applies to `aimxs-customer-hosted` mode (`platform/modes/aimxs-customer-hosted`).
- Covers packaging intake, offline install, offline update, and rollback.
- Assumes AIMXS code remains private and out of OSS control-plane source.

## Required Package Contents

1. Private AIMXS provider runtime artifact:
   - Preferred: OCI image reference + immutable digest.
   - Optional fallback: private signed artifact path for offline bundle ingestion.
2. Signing evidence:
   - signature verification report/reference.
   - attestation/provenance reference.
3. SBOM evidence:
   - SPDX or CycloneDX reference.
4. Customer overlay bundle:
   - `ExtensionProvider` registration values.
   - TLS/bearer secret wiring instructions.
5. Support and SLA references:
   - support boundary document.
   - SLA commitment document.

## Install Flow (Air-Gapped)

1. Import package into customer offline artifact registry/bundle store.
2. Verify digest integrity and signature evidence before deploy.
3. Apply customer-local mode manifests:
   - `kubectl apply -k platform/modes/aimxs-customer-hosted`
4. Configure secure auth material:
   - `auth.mode=MTLS` or `MTLSAndBearerTokenSecret`
   - customer-managed cert/token secrets.
5. Apply runtime network restrictions for no external egress.
6. Validate provider discovery:
   - `Ready=True`, `Probed=True` on AIMXS provider registration.
7. Run local no-egress verification against runtime policy path.

## Update Flow (Air-Gapped)

1. Import new signed package version into offline registry/store.
2. Validate digest/signature/SBOM references.
3. Stage update in non-prod customer namespace.
4. Apply updated AIMXS provider registration with immutable version reference.
5. Execute runtime verification gates:
   - deployment mode switching.
   - entitlement deny path.
   - no-egress behavior.
6. Promote to production only after strict gate pass evidence is archived.

## Rollback

1. Repoint policy selection to previous known-good AIMXS package ref.
2. Keep current package deployed but disabled for forensics if needed.
3. If rollback fails, route policy selection to OSS provider path (`oss-only` mode).
4. Re-run strict runtime verification and capture incident evidence.

## No External Egress Guardrail

- Customer-hosted mode is treated as in-cluster local policy execution.
- Runtime and policy-provider paths must not require outbound internet to succeed.
- Any external dependency must be declared explicitly and reviewed by customer security.

## Evidence Requirements

- Packaging evidence is generated by:
  - `platform/local/bin/verify-m10-customer-hosted-packaging.sh`
- Inputs are sourced from:
  - `../EPYDIOS_AI_CONTROL_PLANE_NON_GITHUB/provenance/aimxs/customer-hosted-release-inputs.vars` (default)
- Evidence artifacts:
  - `../EPYDIOS_AI_CONTROL_PLANE_NON_GITHUB/provenance/aimxs/m10-7-customer-hosted-packaging-evidence-<timestamp>.json`
  - `../EPYDIOS_AI_CONTROL_PLANE_NON_GITHUB/provenance/aimxs/m10-7-customer-hosted-packaging-evidence-<timestamp>.json.sha256`
  - `../EPYDIOS_AI_CONTROL_PLANE_NON_GITHUB/provenance/aimxs/m10-7-customer-hosted-packaging-evidence-latest.json`
